# 版本号与发布管理规范

> 状态：正式规范
> 最后校准：2026-07-21
> 当前软件版本：0.1.0-alpha.1
> 当前存档 Schema 版本：1
> 当前内容版本：2026.07.1

## 一、目标

本规范统一政途人生的：

- 软件版本号
- 开发阶段名称
- Git 分支名称
- Commit 信息
- Pull Request 标题
- Release 与 Tag
- 存档 Schema 版本
- 游戏内容版本

从本规范生效后，不再使用"v3""v4"等名称指代架构代际、设计方案或开发阶段。历史文档中的旧称呼可以保留，但必须标记为历史设计文档。

## 二、版本号格式

项目采用语义化版本：

```
MAJOR.MINOR.PATCH[-PRERELEASE]
```

示例：`0.1.0-alpha.1`、`0.2.0-beta.1`、`1.0.0`

## 三、主版本号规则

MAJOR 表示重大兼容性变化。只有项目正式进入稳定阶段后才使用主版本递增。

- `1.0.0` 表示核心玩法稳定、存档兼容政策明确、主要用户流程完整。
- `2.0.0` 及以后用于无法兼容旧存档的重大模型重构。

### 0.x 阶段

当前项目处于 0.x 阶段，采用比标准 SemVer 更严格的内部规则：

- MINOR 增加表示新增阶段性能力或计划内结构变化
- PATCH 不得引入有意的存档破坏
- 已发布的 alpha/beta 存档如无法迁移，必须明确提示并保留备份
- 不得因为版本小于 1.0 就完全忽略兼容性

## 四、次版本号规则

MINOR 表示阶段能力发布。以下变化应增加 MINOR：

- 新增一套完整玩法系统
- 引入新的职业发展模型
- 新增公务员职级系统
- 新增岗位机会系统
- 新增 NPC 干部生命周期
- 增加新的主要页面或用户流程
- 对存档结构进行可迁移的系统性升级
- 配置格式发生明显变化

## 五、修订号规则

PATCH 表示兼容性修复：

- 修复行动结算错误
- 修复存档加载错误
- 修复 UI 显示问题
- 增加遗漏的输入校验
- 内部重构但不改变可观察行为

PATCH 版本不得故意破坏当前版本存档、已公开配置格式或用户可见操作流程。

## 六、预发布版本

### alpha

`0.1.0-alpha.1`、`0.1.0-alpha.2`

功能仍在快速变化，数据模型可能调整，主要用于开发验证和内部试玩。存档兼容性有限，但必须安全失败。

### beta

`0.1.0-beta.1`

进入条件：当前次版本主要玩法已完成，核心数据模型基本冻结，主要工作转向缺陷修复和体验完善。

### rc

`0.1.0-rc.1`

功能范围已冻结，只接受发布阻塞缺陷修复。

### 正式版本

`0.1.0`

该次版本计划中的能力已完成并经过验证。正式版本不等于整个游戏完成。

## 七、当前正式基线

当前代码基线定义为 `0.1.0-alpha.1`，含义：

- 项目处于 1.0 之前
- 当前是首个正式管理的开发版本
- 已具有可运行原型
- 已完成存档严格解码、统一时间轴、行动快照和 Store 拆分
- 职务、职级和职业机会模型仍未完成
- 不承诺兼容正式版本体系建立前的裸 PlayerSave 存档（无 SaveEnvelope 封装）

## 八、开发阶段与版本号分离

开发阶段不使用版本号命名。

正确写法：

```
当前开发阶段：职业领域模型重构
目标版本：0.2.0-alpha.1
```

错误写法：

```
v4 基础工程
v4 职业系统
```

Phase 编号只用于路线图管理，不属于软件版本号。

## 九、软件版本、存档版本和内容版本分离

### 1. 软件版本

来源：`package.json`。示例：`0.1.0-alpha.1`

用途：软件发布、构建标识、Git Tag、Release、错误诊断、UI 版本显示。

### 2. 存档 Schema 版本

来源：`src/types/save.ts` 中的 `CURRENT_SCHEMA_VERSION`。示例：`1`

- 使用递增整数
- 只用于存档结构兼容和迁移
- 不与软件版本绑定
- 未来版本存档必须拒绝
- 损坏存档必须安全失败

以下变化才提升 Schema 版本：新增/删除/重命名持久化字段、字段类型变化、状态结构重组。

### 3. 内容版本

来源：`src/types/save.ts` 中的 `CURRENT_CONTENT_VERSION`。格式：`YYYY.MM.REVISION`。示例：`2026.07.1`

用于记录职位配置、行动配置、KPI、事件、数值平衡等内容数据版本。内容版本变化不必然意味着 Schema 升级。

## 十、版本号唯一事实来源

版本号的唯一程序事实来源为 `package.json`。

通过 Vite `define` 在构建时注入 `__APP_VERSION__` 全局常量，界面读取该常量显示版本。不在多个组件中手写版本字符串。

## 十一、Commit 规范

绝大多数提交不写版本号。

正确：

```
feat(career): 建立职务与职级领域模型
fix(save): 保留不兼容存档备份
docs: 校准项目基线
```

错误：

```
feat: v4职业体系
fix: 修复v4存档
```

只有发布提交使用版本号：

```
chore(release): prepare 0.1.0-alpha.2
```

## 十二、PR 标题规范

PR 标题描述具体能力，不使用代际版本。

正确：`feat(career): 建立职务与职级领域模型`
错误：`feat: v4第二阶段`

## 十三、分支命名

分支按任务内容组织，不包含产品代际版本：

```
feat/career-domain-model
fix/save-decoder
refactor/career-state
docs/project-baseline
release/0.1.0-alpha.2
```

## 十四、Git Tag

所有正式构建使用带 v 前缀的 Git Tag：`v0.1.0-alpha.1`、`v1.0.0`

这里的 v 仅表示 Git Tag 中的"version"，不代表过去的"v3设计"或"v4改版"。

## 十五、Release 发布条件

每次创建 Release 前必须完成：

1. 更新 `package.json`
2. 更新 `CHANGELOG.md`
3. 确认 Schema 与内容版本
4. 运行完整 CI
5. 验证新建游戏和存档读写
6. 生成 Git Tag
7. 发布 Release Notes

预发布版本必须在 GitHub Release 中标记为 Pre-release。

## 十六、Changelog 规则

从 `0.1.0-alpha.1` 开始维护 `CHANGELOG.md`。分类：Added / Changed / Fixed / Deprecated / Removed / Save compatibility。

其中 Save compatibility 必须说明：是否提升 Schema 版本、是否支持上一版本存档、迁移失败时如何保留备份。

## 十七、存档兼容政策

### 正式版本体系建立前的旧存档

- 裸 PlayerSave 存档（无 SaveEnvelope 封装）不支持自动迁移
- 不静默加载、不静默覆盖
- 保留只读备份
- 显示明确提示

### 基础工程重构期间的存档

基础工程重构（PR #88）生成的 `schemaVersion: 1` 存档在当前版本仍可加载。解码器仅校验 `schemaVersion`，不校验 `contentVersion`。

### 从 0.1.0-alpha.1 开始的后续存档

- 持久化结构变化时提升 `schemaVersion`
- 支持当前受支持版本向下一 Schema 迁移
- 未来版本存档必须拒绝
- 迁移失败必须保留原始备份
- 不允许通过字段剥离强行加载
- 软件版本升级不必然意味着 Schema 升级
- 内容版本变化不必然意味着 Schema 升级

## 十八、禁止代际式命名

从本规范生效后：

- 禁止使用"v4功能""v4职业系统""v4基础工程"等称呼
- Commit 不得包含产品代际式"v4"
- PR 标题不得用版本代际替代功能描述
- Phase 编号不能代替软件版本

## 十九、历史文档命名处理

历史文档可以保留原文件名（如 `zhengtu-v3-design.md`），但文档顶部必须增加历史标记说明。当前改版指导文档命名为 `docs/CAREER_REDESIGN_GUIDE.md`，不使用 `V4_REDESIGN_GUIDE.md`。
